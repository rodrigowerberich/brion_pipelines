Note, that the value parameter {\ttfamily line} is repeatable, but the flag {\ttfamily -\/-\/ignore} is not. So something like

Actions are executed if a parameter matched an argument string in the command line arguments list. Actions are defined using the following member functions or operators\+:


\begin{DoxyItemize}
\item {\ttfamily parameter\+::call(f)} or {\ttfamily parameter\+::operator () (f)}\+: call a callable entity {\ttfamily f} (function, lambda, custom function object) for each one of the matched argument strings. If {\ttfamily f} accepts exactly one parameter that is convertible from {\ttfamily const char\texorpdfstring{$\ast$}{*}}, the command line argument is passed to it. If the parameter list is empty, it is simply called without argument.
\item {\ttfamily parameter\+::set(target, value)}\+: assign a fixed value to a target object; note that the assignment {\ttfamily target = value;} must be a valid statement
\item {\ttfamily parameter\+::set(target)}\+:
\begin{DoxyItemize}
\item A {\ttfamily bool} target is set to true if the flag/value is present and {\bfseries{left unchanged otherwise}}.
\item A target object of fundamental type {\ttfamily T} ({\ttfamily int}, {\ttfamily long}, {\ttfamily float}, {\ttfamily double}, ...) will be assigned the result of converting the argument string to type {\ttfamily T}.
\item Targets of type {\ttfamily std\+::vector\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}} are appended with a value for each matched argument string. Note that {\ttfamily T} must either be (explicitly) convertible from {\ttfamily const char\texorpdfstring{$\ast$}{*}} or a fundamental type.
\end{DoxyItemize}
\item {\ttfamily operator \texorpdfstring{$<$}{<}\texorpdfstring{$<$}{<}} or {\ttfamily operator \texorpdfstring{$>$}{>}\texorpdfstring{$>$}{>}} assign arg strings to lvalues or call callable entities. Which kind of action will be performed is automatically determined through overload resolution.
\end{DoxyItemize}\hypertarget{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md38}{}\doxysubsubsection{\texorpdfstring{Predefined Functions}{Predefined Functions}}\label{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md38}

By default, the parser tries to identify a command line argument (in that order) as
\begin{DoxyItemize}
\item a single flag
\item a concatenation of multiple, {\itshape joinable} flags (in any order)
\item a concatenation of a {\itshape joinable} flag sequence in the order defined in the CLI code
\item a single value parameter
\item a concatenation of a {\itshape joinable} flag/value sequence in the order defined in the CLI code
\item a concatenation of {\itshape joinable} flags \& values in no particular order
\end{DoxyItemize}

If no match was found, the parser tries the same list again without any restrictions imposed by blocking (positional) parameters, conflicting alternatives, etc. If this leads to any match, an error will be reported. This way, {\itshape potential}, but illegal matches can be found and, e.\+g., conflicting alternatives can be reported.

Consider this CLI\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keyword}{auto}\ cli\ =\ (\ \ \mbox{\hyperlink{namespaceclipp_a49eecb5d433ae8b2a2af78ddc7464618}{option}}(\textcolor{stringliteral}{"{}-\/a"{}}),\ \ \mbox{\hyperlink{namespaceclipp_a49eecb5d433ae8b2a2af78ddc7464618}{option}}(\textcolor{stringliteral}{"{}-\/f"{}})\ \&\ \mbox{\hyperlink{namespaceclipp_a3cecc1b8e2ded82a4e6aff98d1f308a7}{value}}(\textcolor{stringliteral}{"{}filename"{}}),\ \ \mbox{\hyperlink{namespaceclipp_a49eecb5d433ae8b2a2af78ddc7464618}{option}}(\textcolor{stringliteral}{"{}-\/b"{}})\ \ );}

\end{DoxyCode}
 If we give {\ttfamily -\/f -\/b} or {\ttfamily -\/b -\/f -\/a} as command line arguments, an error will be reported, since the value after {\ttfamily -\/f} is not optional.

This behavior is fine for most use cases. But what if we want our program to take any string as a filename, because our filenames might also collide with flag names? It can also be hard to get the "{}grammar"{} of complex parsing expressions right. The following definition for example, contains a subtle pitfall\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keyword}{auto}\ cli\ =\ \mbox{\hyperlink{namespaceclipp_a2016b3961d790070f15fc25873232785}{joinable}}(\mbox{\hyperlink{namespaceclipp_a7de409905bae818ae92c3573c1cd7a6e}{repeatable}}(\ \mbox{\hyperlink{namespaceclipp_a49eecb5d433ae8b2a2af78ddc7464618}{option}}(\textcolor{stringliteral}{"{},"{}})\ ,\ \mbox{\hyperlink{namespaceclipp_a7e646335040860006f05a643abfb4f83}{number}}(\textcolor{stringliteral}{"{}number"{}},\ nums)\ ));}

\end{DoxyCode}
 This will not match arguments like {\ttfamily "{}1,"{}}. This is, because, if the repeat group is \textquotesingle{}hit\textquotesingle{} by any of its child parameters, all non-\/optional parameters must also match within the current \textquotesingle{}repeat cycle\textquotesingle{}. So, if the parser hits the {\ttfamily "{},"{}} it expects to find a number arg as well, because it is blocking (positional) and required. Only after seeing this number can it enter the next repeat cycle. Thus, the argument will not be matched, since joined matches are only valid if no error occured. Well, I didn\textquotesingle{}t find a library that makes building simple command line interfaces simple, yet can also be used to build complex CLIs. Over time, {\itshape clipp} evolved into a domain specific language (in pure C++) that I hope can at least somewhat approach the readability of \href{http://docopt.org}{\texttt{ docopt}} but can leverage the benefits (toolability, etc.) of C++\textquotesingle{}s type system.\hypertarget{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md70}{}\doxysubsubsection{\texorpdfstring{Other libraries (\+Boost, POCO, Qt, adishavit/\+Argh, Taywee/args ... or \textquotesingle{}getopt\textquotesingle{})}{Other libraries (Boost, POCO, Qt, adishavit/Argh, Taywee/args ... or 'getopt')}}\label{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md70}

\item often involve a lot of boilerplate (also for very simple use cases)
\item Some libs might be okay in terms of usability, but don\textquotesingle{}t let you build complex interfaces with nested alternatives, mixed commands \& options, positional values, more than 2 flags per option, etc.
\item I really like ad-\/hoc parsers like \href{https://github.com/adishavit/argh}{\texttt{ Argh}} for their simplicity, but they don\textquotesingle{}t generate usage / man pages and don\textquotesingle{}t allow complex interfaces with error checking.
\item Most libs make it really hard to figure out the resulting command line interface by looking at the code (OK, you be the judge if I did a better job at this...).
\item Some libs come with a ton of dependencies (Qt, Boost). I want a single header file!
\item Some libs require the separation of code related to one command/option. I find this harder to maintain than having everything related to one option in one place.
\item Sometimes flag strings of commands/options have to be repeated several times over.
\item Many libs come with restrictions regarding flag names, formatting, joinability, etc.
\end{DoxyItemize}\hypertarget{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md71}{}\doxysubsubsection{\texorpdfstring{What about $<$a href="{}http\+://docopt.\+org"{} $>$docopt$<$/a$>$?}{What about <a href="http://docopt.org" >docopt</a>?}}\label{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md71}
I like the basic idea, but I don\textquotesingle{}t like redundancy in code, especially if it involves repeating string literals. Docopt generates a command line argument parser from a "{}man page docstring"{}. After parsing, you have to query a dictionary to get values or check for option presence. That means you either have to mention the same flag names twice (in the docstring {\itshape and} in the query) or you have to use string variables which makes the docstring hard to read and kind of defeats the purpose of docopt. Furthermore, the code tends to be littered with string-\/to-\/whatever conversions.

I also wanted the ability to keep {\itshape everything} related to one option/command/value together in the code which I found very useful for programs with lots of command line options. Docopt doesn\textquotesingle{}t let you do that since the interface definition and the code performing any actions is always separated (unless you give up on the "{}beautiful doc page"{} idea at the heart of docopt and build your input string to the parser generator piece-\/by-\/piece).\hypertarget{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md72}{}\doxysection{\texorpdfstring{Design Goals}{Design Goals}}\label{md__2mnt_2c_2Users_2rodri_2Documents_2Brion-Assignment_2brion__pipelines_2components_2clipp_2README_autotoc_md72}

\begin{DoxyItemize}
\item minimize boilerplate
\item simple code for simple use cases
\item good code readability (as far as C++ allows for it)
\item avoid ambiguities
\item eliminate repetitions
\item ability to keep code related to one option/command/value together
\item support many different command line interface conventions
\item support different coding styles, conventions \& tastes
\item value semantics wherever possible
\item do not break clipp\textquotesingle{}s general interface and conventions in the future
